## 1. Decision matrix — which approach per tier

Not every multilingual site needs a library. Pick the simplest approach.

### Decision tree

```
Single-language only?
  YES → No i18n needed. Apply locale formatting (dates, numbers, phone) only.
  NO  ↓

Pure HTML site (Tier 1)?
  YES → Separate .html file per locale + hreflang + language toggle (vanilla JS)
  NO  ↓

Astro site (Tier 2)?
  YES → Astro built-in i18n routing + JSON translation files (zero library)
  NO  ↓

Next.js site (Tier 3)?
  → next-intl (bible-tt pattern) — typed, RSC-compatible, minimal config

React SPA (Vite)?
  → Custom context provider (retail-store pattern) — zero deps, simple
```

### Approach summary

| Tier | Stack | i18n approach | Library |
|------|-------|--------------|---------|
| 1 | Pure HTML | Separate `.html` per locale | None |
| 2 | Astro | `astro:i18n` routing + JSON | None (built-in) |
| 3 | Next.js | URL routing + next-intl | `next-intl` |
| SPA | React/Vite | Custom context provider | None |

**Default for Berlin clients:** Astro + Tailwind v4 with DE + EN. German at `/`, English at `/en/`. Add `/pt-br/` only when the client explicitly serves the Brazilian community. Decide at project start — retrofitting i18n is expensive.

## 2. Supported locales and locale config

### Primary locales for this agency

| Code | Language | Market | Use case |
|------|----------|--------|----------|
| `de` | German | Germany (DE) | All Berlin clients — always the primary language |
| `en` | English | Expats / tourists / international | Standard second language for all Berlin clients |
| `pt-BR` | Brazilian Portuguese | Brazil (BR) | Brazilian community businesses |
| `es` | Spanish | Spain / LatAm | Spanish-speaking clients |

### Locale config — single source of truth per project

Define once. Reference everywhere: URLs, hreflang, date formatting, `<html lang>`, sitemap.

```typescript
// src/i18n/config.ts

export const SUPPORTED_LOCALES = ['de', 'en', 'pt-BR'] as const;
// Adjust per client — not every client needs all locales
export const DEFAULT_LOCALE = 'de' as const;

export type SupportedLocale = typeof SUPPORTED_LOCALES[number];

export interface LocaleConfig {
  code: SupportedLocale;
  name: string;        // English display name
  nativeName: string;  // Name in the language itself
  direction: 'ltr' | 'rtl';
  currency: string;    // ISO 4217
  dateFormat: string;  // Human-readable hint
  numberFormat: Intl.NumberFormatOptions;
  htmlLang: string;    // For <html lang="...">
}

export const LOCALE_CONFIG: Record<SupportedLocale, LocaleConfig> = {
  de: {
    code: 'de',
    name: 'German',
    nativeName: 'Deutsch',
    direction: 'ltr',
    currency: 'EUR',
    dateFormat: 'dd.MM.yyyy',   // 15.04.2026
    numberFormat: { style: 'decimal', minimumFractionDigits: 2, maximumFractionDigits: 2 },
    htmlLang: 'de',
  },
  en: {
    code: 'en',
    name: 'English',
    nativeName: 'English',
    direction: 'ltr',
    currency: 'EUR',            // Berlin context → EUR even for EN
    dateFormat: 'dd MMM yyyy',  // 15 Apr 2026 (unambiguous, works for US and UK)
    numberFormat: { style: 'decimal', minimumFractionDigits: 2, maximumFractionDigits: 2 },
    htmlLang: 'en',
  },
  'pt-BR': {
    code: 'pt-BR',
    name: 'Portuguese (Brazil)',
    nativeName: 'Português (Brasil)',
    direction: 'ltr',
    currency: 'EUR',            // Berlin context → EUR
    dateFormat: 'dd/MM/yyyy',   // 15/04/2026
    numberFormat: { style: 'decimal', minimumFractionDigits: 2, maximumFractionDigits: 2 },
    htmlLang: 'pt-BR',
  },
};

// Type guard — validates untrusted input (URL params, cookies, headers)
export function isValidLocale(locale: string): locale is SupportedLocale {
  return SUPPORTED_LOCALES.includes(locale as SupportedLocale);
}

// Safe resolution — always returns a valid locale, never throws
export function getSafeLocale(locale: string | null | undefined): SupportedLocale {
  if (!locale) return DEFAULT_LOCALE;
  const sanitized = locale.replace(/[^a-zA-Z-]/g, '').slice(0, 10);
  return isValidLocale(sanitized) ? sanitized : DEFAULT_LOCALE;
}
```

**Rules:**
- `SUPPORTED_LOCALES` and `DEFAULT_LOCALE` are the single source of truth — used for routing, hreflang, middleware, and `<html lang>`.
- Never hardcode locale strings anywhere except this config file.
- `DEFAULT_LOCALE` is almost always `'de'` for Berlin-market clients.

## 3. URL structure

### Subdirectory approach — recommended for all multilingual sites

```
https://client.de/           → German (primary — DEFAULT_LOCALE, no prefix)
https://client.de/en/        → English
https://client.de/pt-br/     → Brazilian Portuguese
```

**Why subdirectories, not subdomains or separate domains:**
- Consolidates domain authority (all backlinks benefit the whole site)
- One Vercel project, one DNS record, simpler to manage
- Google's recommended approach for small sites
- Separate domains require building authority from scratch on each

### Locale in URL: lowercase, no region suffix

| Locale code | URL segment |
|-------------|------------|
| `de` (default) | `/` (no prefix) |
| `en` | `/en/` |
| `pt-BR` | `/pt-br/` |

The default locale has no prefix. German at `/`, not `/de/`. Keep primary-market URLs short and clean.

Exception: If the client explicitly wants the primary language prefixed, implement `/de/` and a `/` → `/de/` redirect. Document this choice in the client's CLAUDE.md.

### Locale detection from path

```typescript
export function detectLocaleFromPath(pathname: string): SupportedLocale | null {
  const first = pathname.split('/').filter(Boolean)[0];
  return first && isValidLocale(first) ? first : null;
  // Returns null for default locale (root path, no prefix)
}
```

## 4. Translation file structure and naming

### Reference locale is always the default locale

For Berlin clients: **German (`de`) is the reference locale.** Write German strings first. Write (not translate) English and Portuguese independently. All locale files must have identical key sets.

### File organization

**Simple (single-page or 2-page site):**
```
src/i18n/locales/
├── de.json
├── en.json
└── pt-br.json
```

**Standard (multi-page site):**
```
src/i18n/locales/
├── de/
│   ├── common.json    ← nav, footer, buttons, a11y labels, CTA
│   └── pages.json     ← page-specific content per section
├── en/
│   ├── common.json
│   └── pages.json
└── pt-br/
    ├── common.json
    └── pages.json
```

**Large (Tier 3 Next.js with many pages):**
```
src/i18n/locales/
├── de/
│   ├── common.json
│   ├── home.json
│   ├── services.json
│   └── contact.json
└── en/ (same structure)
```

## 5. Key naming conventions

### Dot-notation namespacing

```
[namespace].[section].[element]

nav.home
nav.services
nav.about
nav.contact
footer.address
footer.phone
footer.hours
hero.headline
hero.subheadline
hero.cta
services.title
services.haircut.name
services.haircut.description
services.haircut.price
contact.hours.weekdays
contact.hours.saturday
faq.booking.question
faq.booking.answer
form.name.label
form.name.placeholder
form.name.error
form.submit
```

### Rules (consistent across all source projects)

- **All key names are in English.** Key names are code. Translated values are in the target language.
- **Hierarchical, not flat.** `hero.headline` not `heroHeadline` or `hero_headline`.
- **Descriptive, not positional.** `nav.home` not `nav.item1`.
- **Identical across all locales.** If `de/common.json` has `footer.address`, every other locale file must also have `footer.address`. Missing keys trigger the fallback.

```json
// de/common.json
{
  "nav": {
    "home": "Startseite",
    "services": "Leistungen",
    "about": "Über uns",
    "contact": "Kontakt"
  },
  "footer": {
    "address": "Musterstraße 1, 10115 Berlin",
    "phone": "+49 30 123 456 78",
    "hours": "Mo–Fr: 9–18 Uhr, Sa: 10–16 Uhr"
  }
}

// en/common.json — same keys, different values
{
  "nav": {
    "home": "Home",
    "services": "Services",
    "about": "About us",
    "contact": "Contact"
  },
  "footer": {
    "address": "Musterstraße 1, 10115 Berlin",
    "phone": "+49 30 123 456 78",
    "hours": "Mon–Fri: 9am–6pm, Sat: 10am–4pm"
  }
}
```

### ICU interpolation for dynamic values

```json
{
  "services": {
    "price": "Ab {price} €",
    "rating": "Bewertet mit {rating} von 5 Sternen",
    "count": "{count, plural, one {# Leistung} other {# Leistungen}}"
  }
}
```

### What never goes in translation files

- URLs and hrefs (code, not content)
- CSS class names
- Raw HTML tags (use component slots or interpolation)
- Conditional logic

## 6. Locale detection and persistence

### Detection priority chain (from diBoaS)

```
1. URL path segment (most reliable — explicit user choice in URL)
   /en/services → locale = 'en'
   /services    → locale = DEFAULT_LOCALE

2. Cookie (user's previous explicit language switch)
   Cookie: LOCALE=en → locale = 'en'

3. Accept-Language header (browser's stated preference)
   Accept-Language: pt-BR,pt;q=0.9,en;q=0.8 → locale = 'pt-BR'

4. Default locale fallback
   → locale = DEFAULT_LOCALE ('de')
```

### Accept-Language parsing (from diBoaS config.ts)

The browser sends quality-weighted preferences. Parse in order, respect q-factor:

```typescript
export function matchAcceptLanguage(header: string | null): SupportedLocale | null {
  if (!header) return null;

  const parsed = header
    .split(',')
    .map(entry => {
      const [lang, ...params] = entry.trim().split(';');
      const qParam = params.find(p => p.trim().startsWith('q='));
      const q = qParam ? parseFloat(qParam.trim().slice(2)) : 1;
      return { lang: lang.trim(), q: isNaN(q) ? 0 : q };
    })
    .sort((a, b) => b.q - a.q);

  for (const { lang } of parsed) {
    if (isValidLocale(lang)) return lang;                      // Exact: "pt-BR"
    const prefix = lang.split('-')[0];                        // Language-only: "pt" → "pt-BR"
    const match = SUPPORTED_LOCALES.find(
      l => l === prefix || l.startsWith(`${prefix}-`)
    );
    if (match) return match;
  }

  return null;
}

export function detectPreferredLocale(
  cookieLocale: string | null | undefined,
  acceptLanguage: string | null | undefined
): SupportedLocale {
  if (cookieLocale && isValidLocale(cookieLocale)) return cookieLocale;
  const matched = matchAcceptLanguage(acceptLanguage ?? null);
  return matched ?? DEFAULT_LOCALE;
}
```

### Language persistence

When the user manually switches language, persist the choice:

```typescript
// Persist to localStorage (SPA) or set a cookie (SSR)
function setLocale(locale: SupportedLocale) {
  try { localStorage.setItem('locale', locale); } catch {}
  // For URL-based routing: navigate to the new locale URL
  const prefix = locale === DEFAULT_LOCALE ? '' : `/${locale}`;
  window.location.href = `${prefix}${currentPath}`;
}

// Safe read (wrapped for SSR compatibility)
function getStoredLocale(): SupportedLocale | null {
  try {
    const stored = localStorage.getItem('locale');
    return stored && isValidLocale(stored) ? stored as SupportedLocale : null;
  } catch {
    return null;
  }
}
```

- **Tier 1 (HTML):** Store in `localStorage`. On load, check and redirect if needed.
- **Tier 2 (Astro) and Tier 3 (Next.js):** URL is the source of truth. Persist in a cookie as a hint for middleware on the next visit.

### Language switcher component

Every multilingual site needs one. Place in the header. Show **native language name**, not the English name.

```html
<nav aria-label="Sprache wählen / Language">
  <a href="/" hreflang="de" lang="de" aria-current="true">Deutsch</a>
  <a href="/en/" hreflang="en" lang="en">English</a>
  <a href="/pt-br/" hreflang="pt-BR" lang="pt-BR">Português</a>
</nav>
```

**Rules:**
- Native names: `Deutsch`, `English`, `Português` — not `German`, `Inglês`, `Alemão`
- Mark current language with `aria-current="true"`
- Each link includes `lang` (target language) and `hreflang` (search engine signal)
- Never use flags as the sole indicator. 🇧🇷 ≠ all Portuguese speakers. Flags as supplementary decoration are fine.

## 7. Fallback chain

When a key is missing in the current locale:

```
1. Current locale file      → de/common.json: "nav.home": "Startseite" ✓
2. Default locale file      → fall back to de/ if current locale is not 'de'
3. Return the key string    → "nav.home" renders in the UI (visible bug → fix it)
```

**From retail-store's `getNestedValue`:**

```typescript
function getNestedValue(obj: Record<string, unknown>, path: string): string {
  const keys = path.split('.');
  let current: unknown = obj;
  for (const key of keys) {
    if (current && typeof current === 'object' && key in (current as Record<string, unknown>)) {
      current = (current as Record<string, unknown>)[key];
    } else {
      return path; // Returns the key itself — visible in UI, signals a missing translation
    }
  }
  return typeof current === 'string' ? current : path;
}
```

**From diBoaS's `loadMessages`:**

```typescript
async function loadMessages(locale: SupportedLocale, namespace: string) {
  try {
    return await import(`../locales/${locale}/${namespace}.json`);
  } catch {
    if (locale !== DEFAULT_LOCALE) {
      try {
        return await import(`../locales/${DEFAULT_LOCALE}/${namespace}.json`);
      } catch {
        return {};
      }
    }
    return {};
  }
}
```

**Rules:**
- A key falling back to itself is a development safety net, not a production feature. It must be fixed.
- Run key parity validation before every deploy (see Section 14 for the script).
- Never release with visible translation keys (raw dot-notation strings) in the UI.

## 8. Date, number, and currency formatting

### Always use the Intl API — never format manually

```typescript
const date = new Date('2026-04-15');

// Date formatting
new Intl.DateTimeFormat('de-DE').format(date)           // "15.4.2026"
new Intl.DateTimeFormat('en-US').format(date)           // "4/15/2026"
new Intl.DateTimeFormat('pt-BR').format(date)           // "15/04/2026"

// Long format (readable, unambiguous)
new Intl.DateTimeFormat('de-DE', { dateStyle: 'long' }).format(date)  // "15. April 2026"
new Intl.DateTimeFormat('en-GB', { dateStyle: 'long' }).format(date)  // "15 April 2026"
new Intl.DateTimeFormat('pt-BR', { dateStyle: 'long' }).format(date)  // "15 de abril de 2026"
```

### Date format reference

| Locale | Short format | Long format example |
|--------|-------------|---------------------|
| `de` | `dd.MM.yyyy` | `15. April 2026` |
| `en` | `dd MMM yyyy` (unambiguous) | `15 Apr 2026` |
| `pt-BR` | `dd/MM/yyyy` | `15 de abril de 2026` |

**For Berlin-market EN sites:** Use `dd MMM yyyy` (e.g., `15 Apr 2026`). It's unambiguous and works for both US and UK readers.

### Opening hours — static strings in translation files

Hours on local business sites are static, not computed. Store in translation files, not generated with Intl:

```json
// de/pages.json
{
  "hours": {
    "weekdays": "Montag–Freitag: 9:00–18:00 Uhr",
    "saturday": "Samstag: 10:00–16:00 Uhr",
    "sunday": "Sonntag: geschlossen"
  }
}

// en/pages.json
{
  "hours": {
    "weekdays": "Monday–Friday: 9am–6pm",
    "saturday": "Saturday: 10am–4pm",
    "sunday": "Sunday: closed"
  }
}
```

German uses 24-hour clock (`18:00 Uhr`). English uses 12-hour clock (`6pm`). These are conventions, not just word translations.

### Number and currency formatting

```typescript
// Prices — always locale-aware, never hardcoded
new Intl.NumberFormat('de-DE', { style: 'currency', currency: 'EUR' }).format(45)
// → "45,00 €"

new Intl.NumberFormat('en-US', { style: 'currency', currency: 'EUR' }).format(45)
// → "€45.00"

new Intl.NumberFormat('pt-BR', { style: 'currency', currency: 'EUR' }).format(45)
// → "€ 45,00"
```

**Critical difference:** German uses comma as decimal separator and period as thousands separator (`1.234,56 €`). English is the opposite (`€1,234.56`). Never hardcode `€45.00` on a German page.

### Phone numbers

Not formatted with Intl. Use international format everywhere on multilingual sites:

```
+49 30 123 456 78   ← International format — correct on all locale versions
030 123 456 78      ← Local format — only obvious to people who know German area codes
```

A Brazilian reader on the English version of the site doesn't know `030` is Berlin. Use `+49`.

## 9. Text expansion budgets

### Why this matters

Translated text is longer than the source. Layouts that don't account for this break when language is switched.

| From → To | Expected expansion |
|-----------|-------------------|
| English → German | +30% to +40% |
| English → Brazilian Portuguese | +20% to +30% |
| German → English | −20% to −25% |
| German → Brazilian Portuguese | approximately equal |

### Every component must be tested in the longest language before delivery

For Berlin-market sites, German is typically the longest. Test at +40% — if it doesn't break at the extreme, the others will be fine.

### Common problem areas

| UI element | Problem | Fix |
|------------|---------|-----|
| Navigation items | `"Über uns"` > `"About"` | Use `white-space: nowrap` + responsive nav, or test at DE width |
| Buttons | `"Jetzt Termin buchen"` > `"Book now"` | `width: fit-content` + `min-width`, never fixed width |
| Form labels | German labels are consistently longer | Leave extra width, or stack label above input |
| Mobile nav | Long German labels may push items off-screen | Test at 375px in German |
| Card titles | German compound words don't hyphenate by default | Add `word-break: break-word` to base styles |

### CSS baseline to add to every multilingual project

```css
/* Prevent layout breaks from long German compound words */
body {
  overflow-wrap: break-word;
  word-break: break-word;
}

/* Buttons: grow, don't clip */
button, .btn, a[role="button"] {
  white-space: normal;
  min-width: fit-content;
}
```

## 10. Implementation patterns by stack

### Tier 1 — Pure HTML

Separate HTML file per locale. Language switcher with direct links. No build step.

```
/index.html          → German (primary)
/en/index.html       → English
/pt-br/index.html    → Brazilian Portuguese
```

```html
<!-- In <head> of every page -->
<link rel="alternate" hreflang="de" href="https://client.de/" />
<link rel="alternate" hreflang="en" href="https://client.de/en/" />
<link rel="alternate" hreflang="pt-BR" href="https://client.de/pt-br/" />
<link rel="alternate" hreflang="x-default" href="https://client.de/" />

<!-- Language switcher in <header> -->
<nav aria-label="Sprache wählen">
  <a href="/" hreflang="de" lang="de">Deutsch</a>
  <a href="/en/" hreflang="en" lang="en">English</a>
  <a href="/pt-br/" hreflang="pt-BR" lang="pt-BR">Português</a>
</nav>
```

**Pros:** Zero complexity, best performance, works on any host.
**Cons:** Content sync is manual. Suitable for 1–3 page sites only.

### Tier 2 — Astro

```typescript
// astro.config.ts
import { defineConfig } from 'astro/config';

export default defineConfig({
  site: 'https://client.de',
  i18n: {
    defaultLocale: 'de',
    locales: ['de', 'en', 'pt-br'],
    routing: {
      prefixDefaultLocale: false,  // / = German, /en/ = English
    },
  },
});
```

**Translation utility:**

```typescript
// src/i18n/utils.ts
import type { SupportedLocale } from './config';

type Messages = Record<string, unknown>;

function getNestedValue(obj: Messages, path: string): string {
  const keys = path.split('.');
  let current: unknown = obj;
  for (const key of keys) {
    if (current && typeof current === 'object' && key in (current as Messages)) {
      current = (current as Messages)[key];
    } else {
      return path; // Visible fallback — means missing key
    }
  }
  return typeof current === 'string' ? current : path;
}

export async function useTranslations(locale: SupportedLocale) {
  const messages = (await import(`./locales/${locale}.json`)).default as Messages;
  return (key: string) => getNestedValue(messages, key);
}
```

**Usage in Astro pages:**

```astro
---
// src/pages/en/index.astro
import BaseLayout from '../../layouts/BaseLayout.astro';
import { useTranslations } from '../../i18n/utils';

const t = await useTranslations('en');
---

<BaseLayout locale="en" title={t('seo.title')} description={t('seo.description')}>
  <h1>{t('hero.headline')}</h1>
  <p>{t('hero.subheadline')}</p>
  <a href={`tel:+49...`}>{t('hero.cta')}</a>
</BaseLayout>
```

**BaseLayout handles hreflang automatically:**

```astro
---
// src/layouts/BaseLayout.astro
import { SUPPORTED_LOCALES, DEFAULT_LOCALE, LOCALE_CONFIG } from '../i18n/config';

interface Props {
  locale: string;
  title: string;
  description: string;
}

const { locale, title, description } = Astro.props;
const canonicalUrl = Astro.site?.href + (locale === DEFAULT_LOCALE ? '' : `${locale}/`);
---

<html lang={LOCALE_CONFIG[locale]?.htmlLang ?? locale}>
<head>
  <title>{title}</title>
  <meta name="description" content={description} />
  <link rel="canonical" href={canonicalUrl} />

  {SUPPORTED_LOCALES.map(l => (
    <link
      rel="alternate"
      hreflang={LOCALE_CONFIG[l].htmlLang}
      href={Astro.site?.href + (l === DEFAULT_LOCALE ? '' : `${l}/`)}
    />
  ))}
  <link rel="alternate" hreflang="x-default" href={Astro.site?.href} />
</head>
```

### SPA — Custom React context (retail-store pattern)

For React + Vite projects (no SSR). Zero dependencies. Clean and maintainable.

```typescript
// src/i18n/index.tsx
import { createContext, useContext, useState, useCallback, type ReactNode } from 'react';
import de from './locales/de.json';
import en from './locales/en.json';
import ptBR from './locales/pt-br.json';

export type Locale = 'de' | 'en' | 'pt-br';

const messages: Record<Locale, Record<string, unknown>> = {
  de, en, 'pt-br': ptBR,
};

const browserLocales: Record<Locale, string> = {
  de: 'de-DE', en: 'en-US', 'pt-br': 'pt-BR',
};

function getNestedValue(obj: Record<string, unknown>, path: string): string {
  const keys = path.split('.');
  let current: unknown = obj;
  for (const key of keys) {
    if (current && typeof current === 'object' && key in (current as Record<string, unknown>)) {
      current = (current as Record<string, unknown>)[key];
    } else {
      return path;
    }
  }
  return typeof current === 'string' ? current : path;
}

function getInitialLocale(): Locale {
  try {
    const stored = localStorage.getItem('locale') as Locale;
    if (stored && stored in messages) return stored;
    const browser = navigator.language.toLowerCase();
    if (browser.startsWith('de')) return 'de';
    if (browser.startsWith('pt')) return 'pt-br';
    if (browser.startsWith('en')) return 'en';
  } catch {}
  return 'de';
}

const I18nContext = createContext<{
  locale: Locale;
  setLocale: (l: Locale) => void;
  t: (key: string) => string;
  intlLocale: string;
} | null>(null);

export function I18nProvider({ children }: { children: ReactNode }) {
  const [locale, setLocaleState] = useState<Locale>(getInitialLocale);

  const setLocale = useCallback((newLocale: Locale) => {
    try { localStorage.setItem('locale', newLocale); } catch {}
    setLocaleState(newLocale);
  }, []);

  const t = useCallback(
    (key: string) => getNestedValue(messages[locale], key),
    [locale]
  );

  return (
    <I18nContext value={{ locale, setLocale, t, intlLocale: browserLocales[locale] }}>
      {children}
    </I18nContext>
  );
}

export function useTranslation() {
  const ctx = useContext(I18nContext);
  if (!ctx) throw new Error('useTranslation must be inside I18nProvider');
  return ctx;
}
```

## 11. hreflang and SEO wiring

### Required on every page of every multilingual site

See `SEO.md` Section 7 for the complete specification. Key rules:

- Every page must declare ALL language versions, including itself (self-referencing is required)
- The relationship is symmetric: if `/de` links to `/en`, then `/en` must link back to `/de`
- Always include `x-default` pointing to the primary locale (or a language selector page)
- Never use IP-based auto-redirect — Google's bot can't crawl all versions

### HTML lang attribute

Must match the current locale's BCP 47 code — note the different casing from URL slugs:

| URL segment | `<html lang>` value |
|------------|---------------------|
| `/` (German) | `lang="de"` |
| `/en/` | `lang="en"` |
| `/pt-br/` | `lang="pt-BR"` (capital B and R — BCP 47 standard) |

### Locale-specific Open Graph metadata

```html
<meta property="og:locale" content="de_DE" />          <!-- German page -->
<meta property="og:locale:alternate" content="en_US" />
<meta property="og:locale:alternate" content="pt_BR" />
```

The `og:locale` format uses underscore: `de_DE`, `en_US`, `pt_BR` — not hyphens.

### Programmatic hreflang generation

```typescript
// src/i18n/config.ts
export function generateAlternateUrls(
  pagePath: string,
  siteUrl: string,
  locales: SupportedLocale[] = [...SUPPORTED_LOCALES]
): Record<string, string> {
  const alternates: Record<string, string> = {};
  for (const locale of locales) {
    const prefix = locale === DEFAULT_LOCALE ? '' : `/${locale}`;
    const normalizedPath = pagePath === '/' ? '' : pagePath;
    alternates[locale] = `${siteUrl}${prefix}${normalizedPath}`;
  }
  alternates['x-default'] = `${siteUrl}${pagePath === '/' ? '' : pagePath}`;
  return alternates;
}
```

## 12. German-specific rules

### Formal vs informal address — choose one, stay consistent

German has `Sie` (formal) and `du` (informal). The choice affects verb conjugations across the entire site.

| Use `Sie` (formal) | Use `du` (informal) |
|-------------------|---------------------|
| Professional services (doctor, lawyer, accountant) | Creative/lifestyle (salon, tattoo, yoga, café) |
| Older clientele | Younger clientele |
| B2B businesses | B2C leisure businesses |

**Choose once. Apply everywhere. Never mix on the same site.**

### German typography

- Quotation marks: „German style" (not "English style")
- Dashes for ranges: `–` (en dash) — `9–18 Uhr` not `9-18 Uhr`
- Decimal separator: `,` (comma) — `45,00 €`
- Thousands separator: `.` (period) — `1.000`
- Apostrophes: Not used for possessives in German

### German capitalization

All nouns are capitalized in German. This is grammar, not style. Failing to capitalize nouns is immediately visible to native speakers.

```
CORRECT: "Unser Friseur bietet professionelle Haarschnitte und Farben."
WRONG:   "Unser friseur bietet professionelle haarschnitte und farben."
```

### Compound words and line breaking

German creates long compound words (`Öffnungszeiten`, `Terminvereinbarung`, `Kostenvoranschlag`) that don't break by default in browsers. Add to base styles:

```css
body {
  overflow-wrap: break-word;
  word-break: break-word;
  hyphens: auto;  /* Requires lang="de" on the HTML element */
}
```

### Impressum — legally required

Every German-market website must have an Impressum page, reachable within two clicks from any page (typically footer link). Required by German Telemediengesetz (TMG).

**Minimum Impressum content:**
- Full legal name of the business owner
- Physical street address (PO box is insufficient)
- Email address
- Phone number
- Handelsregisternummer if GmbH/UG/AG
- USt-IdNr. if VAT registered
- Verantwortlich für den Inhalt (responsible person for editorial content)

Generate using eRecht24 or IT-Recht Kanzlei. Have the client verify before going live.

## 13. Portuguese-specific rules

### pt-BR vs pt-PT — they are different

Never use European Portuguese translations for a Brazilian audience or vice versa. Key differences:

| Topic | pt-BR | pt-PT |
|-------|-------|-------|
| Second person | `você` + third-person verb | `tu` + second-person verb |
| Vocabulary | Many anglicisms | More conservative |
| Gerund | `estou fazendo` | `estou a fazer` |
| Placement of pronouns | Different rules | Different rules |

Most Brazilian community businesses in Berlin serve a Brazilian audience → use `pt-BR`.

### pt-BR typography

- Quotation marks: "Modern style" (same as English) — or „German style" for formal documents
- Decimal separator: `,` (comma) — `45,00 €`
- Thousands separator: `.` (period) — `1.000`
- Date: `dd/MM/yyyy` — `15/04/2026`

### pt-BR tone

Brazilian Portuguese tends toward a warmer, more casual register than European Portuguese. Match the overall tone of the client's brand (formal vs informal) but don't write stiffly — Brazilian readers notice.

## 14. Translation workflow

### Sequence for every new multilingual client

```
1. Write primary language content (German for Berlin clients)
   → All section copy, FAQs, buttons, nav items, meta tags, hours, prices

2. Audit primary language for completeness
   → Every translation key populated?
   → Every page has unique title and description?
   → Prices, hours, contact info confirmed with client?

3. Write secondary language content — INDEPENDENTLY, not as translation
   → English version: written fresh for an English-speaking audience
   → PT-BR version: written fresh for a Brazilian audience
   → Same meaning, adapted phrasing, different sentence structure if needed

4. Run key parity check
   → Every key in de.json must exist in en.json and pt-br.json
   → No extra keys, no missing keys
   → Run validate:translations script before every deploy

5. Test layout in German (longest language)
   → Check every button, nav item, label for overflow
   → Test at 375px mobile width

6. Test locale switching
   → Switch between all locales on every page
   → Verify: HTML lang changes, hreflang is correct, all dynamic content renders

7. Client approval
   → Client reviews and approves all language versions in their language
   → This is non-negotiable
```

### Machine translation policy

- **Never use machine translation as the final output.** It produces errors and unnatural phrasing.
- **Machine translation as a first draft is acceptable.** Use DeepL (best for DE/EN/PT-BR). Then review and rewrite every sentence.
- **Client must approve all translations before going live.** Their business, their reputation.

### Key parity validation script

Run before every deploy to catch missing translations:

```typescript
// scripts/validate-translations.ts
import de from '../src/i18n/locales/de.json';
import en from '../src/i18n/locales/en.json';
import ptBR from '../src/i18n/locales/pt-br.json';

function getKeys(obj: unknown, prefix = ''): string[] {
  if (typeof obj !== 'object' || obj === null) return [prefix];
  return Object.entries(obj as Record<string, unknown>).flatMap(([k, v]) =>
    getKeys(v, prefix ? `${prefix}.${k}` : k)
  );
}

const deKeys = new Set(getKeys(de));
const enKeys = new Set(getKeys(en));
const ptKeys = new Set(getKeys(ptBR));

let errors = 0;

for (const key of deKeys) {
  if (!enKeys.has(key)) { console.error(`MISSING in en: ${key}`); errors++; }
  if (!ptKeys.has(key)) { console.error(`MISSING in pt-br: ${key}`); errors++; }
}
for (const key of enKeys) {
  if (!deKeys.has(key)) { console.error(`EXTRA in en (not in de): ${key}`); errors++; }
}

if (errors > 0) {
  console.error(`\n${errors} parity error(s). Fix before deploying.`);
  process.exit(1);
} else {
  console.log('All translation keys match.');
}
```

```json
// package.json
{
  "scripts": {
    "validate:translations": "tsx scripts/validate-translations.ts",
    "build": "pnpm validate:translations && astro build"
  }
}
```

## 15. Quality checklist

Run before every multilingual delivery.

### Structure

- [ ] `SUPPORTED_LOCALES` and `DEFAULT_LOCALE` in one config file
- [ ] URL structure: default locale at `/`, others at `/en/`, `/pt-br/`
- [ ] Language switcher in header on every page
- [ ] Native language names shown (Deutsch, English, Português)

### Correctness

- [ ] `<html lang>` matches current locale on every page
- [ ] `hreflang` tags present and symmetric on every page
- [ ] `x-default` hreflang points to the default locale URL
- [ ] `og:locale` matches current locale on every page
- [ ] Key parity check passes (validate:translations script)
- [ ] No raw translation keys visible in the UI on any locale

### Content

- [ ] All content written in the target language, not machine-translated verbatim
- [ ] Client has reviewed and approved all translations
- [ ] Phone number in international format (+49...) on all locale versions
- [ ] Opening hours formatted correctly per locale (24h for DE, 12h for EN)

## 16. Anti-patterns

### Architecture

- **Translating code into the target language.** `const Öffnungszeiten = ...` or `const horasFuncionamento = ...` in source code. Code is English. Always.
- **Hardcoding any visible string in a component on a multilingual site.** Every user-facing string goes through the translation system.
- **One JSON file shared across all locales.** Each locale gets its own file.
- **Adding i18n as an afterthought.** Decide at the start. Retrofitting is expensive.
- **Multiple i18n systems in one project.** One system. If the project uses next-intl, everything uses next-intl.

### Content

- **Word-for-word machine translation as final output.** Always review, rewrite, and get client approval.
- **Translating English keyword research as German/Portuguese SEO targets.** Research each locale independently (SEO.md Section 7).
- **Flags as the sole language indicator.** Text labels are required.
- **Different tone on different locale versions** (formal on DE, informal on EN). Pick one brand voice and apply it consistently.
- **Missing Impressum on German-market sites.** Legal requirement. Non-negotiable.

### SEO

- **Missing hreflang.** Google serves the wrong language version to the wrong audience.
- **Non-symmetric hreflang.** If `/de/` links to `/en/`, then `/en/` must link to `/de/`. Google ignores asymmetric hreflang.
- **Missing `x-default`.** Required. Points to the primary locale.
- **IP-based auto-redirect.** Never. Google can't crawl all versions.
- **Duplicate content across locales.** Each locale version must have distinct, non-identical content.

### Technical

- **Reading `localStorage` during SSR without a guard.** Always wrap in `try/catch` or check `typeof window !== 'undefined'`.
- **Inconsistent locale code format.** `pt-BR` in hreflang and `<html lang>`, `pt-br` in URLs and file names. Never mix these within the same layer.
- **Using `pt_BR` (underscore) in URLs.** Underscores in URLs are treated as word separators by some crawlers. Use hyphens in URLs.
- **Leaking default locale into the URL** (`/de/` when it should be `/`). Unless explicitly chosen.
